# Future Plan: Real-Time Delivery Partner Metrics Dashboard

## What we are seeking
The business needs a real-time view to monitor live delivery operations, specifically showing active deliveries in transit, delivery attempt counts, and SLA breaches. Crucially, this must be grouped by the actual Delivery Partner Name (e.g., "FedEx", "Delhivery") instead of just the raw Partner ID.

## How it can be done directly in BI/Reporting Tools (No new tables)
Because we use Delta Lake, reporting tools (like Tableau, PowerBI, or Databricks SQL) can directly query the Silver layer without us needing to build a dedicated Gold table for this.
The BI tool simply runs a live SQL query that joins the fast-moving streaming facts with the slow-moving static dimensions:
- JOIN `silver_base.delivery_fact_RT` (streaming) WITH `silver_enriched.delivery_partner_dim` (batch SCD2) ON `partner_id`.
- Filter for `is_current = true` on the dimension and `actual_delivery_time IS NULL` on the fact.
- Group by `partner_name`.

## What we need to do if we want to build a dedicated Gold Table (#28)
If the live BI join proves too resource-heavy or slow, we can pre-compute the dashboard in Spark by adding a 5th Gold Semantic RT table (`gold_delivery_status_RT`):
1. In Spark, use `readStream` on `silver_delivery_fact_RT`.
2. Perform a "Stream-Static Join" against the latest version of `silver_delivery_partner_dim`.
3. Aggregate the data (e.g., using a 5-minute window grouped by `partner_name`).
4. Use `writeStream` to push the aggregated results into the new Gold Delta table continuously.

## 6. Dead Letter Queue (DLQ) for All Streaming Layers
After all table development is complete, add good/bad data isolation to every streaming write.
Streams must NEVER fail — bad data should be redirected, not dropped.

### Tables that need DLQ:
- Landing stream (raw Kafka → Landing Delta)
- Bronze stream (Landing → Bronze Delta)
- Silver Base: order_fact_RT, order_event_fact_RT, order_item_fact_RT
- Silver Base: payment_attempt_fact_RT, delivery_fact_RT, discount_fact_RT
- Gold RT: sales_summary_RT, category_sales_metrics_RT
- Gold RT: payment_usage_metrics_RT, order_lifecycle_metrics_RT

### Implementation pattern for each:
Every streaming table gets table-specific validation rules (not just null checks).
Bad rows get a `rejection_reason` column so ops can instantly diagnose failures.

```python
# 1. Define table-specific rules
good_df = transformed_df.filter("order_id IS NOT NULL AND price > 0 AND quantity > 0")

# 2. Tag bad rows with WHY they failed
bad_df = transformed_df.withColumn("rejection_reason",
    concat_ws(", ",
        when(col("order_id").isNull(), lit("missing_order_id")),
        when(col("price") <= 0,        lit("invalid_price")),
        when(col("quantity") <= 0,     lit("invalid_quantity"))
    )
).filter("rejection_reason != ''")

good_df.writeStream...  # → actual Silver/Gold table
bad_df.writeStream...   # → dead_letter/<layer_name> table
```

### DLQ monitoring:
- Create a monitoring job that checks DLQ row counts every 5 minutes
- If DLQ count spikes → fire alert to Slack / PagerDuty

## 7. Duplicate Detection
Kafka can redeliver the same message (producer retry, consumer rebalance).
Without dedup, the same order gets counted twice in Gold dashboards.

### Where to deduplicate:
- Silver Base: Do NOT deduplicate here (append-only for speed)
- Gold Batch (fact_orders, fact_order_items): Deduplicate using ROW_NUMBER
  ```sql
  SELECT * FROM (
      SELECT *, ROW_NUMBER() OVER (PARTITION BY order_id ORDER BY bronze_ingest_ts) AS rn
      FROM silver_base.order_fact_RT
  ) WHERE rn = 1
  ```
- Gold RT: Use `dropDuplicatesWithinWatermark(["order_id"])`

## 8. Watermarking for Gold RT Tables
Only the 4 Gold RT tables need watermarking (they use 5-min tumbling windows).
Without watermark, Spark keeps every window open forever → memory grows infinitely.

### Tables that need watermarking:
- gold_sales_summary_RT
- gold_category_sales_metrics_RT
- gold_payment_usage_metrics_RT
- gold_order_lifecycle_metrics_RT

### Implementation:
```python
df.withWatermark("event_time", "30 minutes") \
  .groupBy(window("event_time", "5 minutes"), "store_id", "category") \
  .agg(...)
```
Late data arriving within 30 minutes of the window is accepted.
Data arriving after 30 minutes is dropped (already in Silver for backfill if needed).

## 9. Data Freshness / Pipeline SLA Monitoring
A separate monitoring job (not inside any table logic) that runs every 5 minutes.
Checks `MAX(ingest_timestamp)` at each layer. If any layer is stale beyond threshold → alert.

### Implementation:
```sql
SELECT 'Landing' AS layer, MAX(landing_ingest_ts) AS last_data FROM landing.raw_events
UNION ALL
SELECT 'Bronze',  MAX(bronze_ingest_ts)  FROM bronze.orders_raw
UNION ALL
SELECT 'Silver',  MAX(bronze_ingest_ts)  FROM silver_base.order_fact_RT
UNION ALL
SELECT 'Gold RT', MAX(window_end)        FROM gold_semantic.sales_summary_RT
```
Threshold: If any layer is >5 minutes stale → alert ops.

## 10. Row Count Reconciliation
Verify zero data loss across layers after every batch cycle.
Landing count = Bronze count = Silver good + Silver DLQ.
If numbers don't add up, something was silently dropped.

# Data Quality Gate Strategy (Structured Reference)

## Gate 0: Schema Registry (Kafka)
- Blocks structural corruption BEFORE data enters Kafka
- Prevents: removed fields, type changes, broken JSON structure
- Does NOT catch: null values, invalid business values, junk data

## Gate 1: DLQ at Silver Base (Streaming)
- Splits every micro-batch into good + bad BEFORE writing to Silver
- Bad rows get a `rejection_reason` column for ops diagnosis
- Checks applied:
  - Null primary keys (order_id, customer_id, item_id)
  - Business rules (price > 0, quantity > 0, valid payment_method)
  - Format rules (zip = 6 digits, phone = 10 digits)
- Good → Silver table. Bad → Dead letter table. Stream NEVER stops.

## Gate 2: Assertions at Silver Enriched (Batch)
- Validates incoming data BEFORE MERGE into SCD2 dimension tables
- If validation fails → job crashes → table stays UNTOUCHED → fix and rerun
- Why: Bad data in SCD2 tables is extremely hard to remove (historical versioning)
- Checks: null business keys, invalid segments, malformed attributes

## Gate 3: DLQ at Gold RT (Streaming)
- Same good/bad split pattern as Silver
- Additional: dropDuplicatesWithinWatermark to remove Kafka redeliveries
- Additional: withWatermark("event_time", "30 minutes") for late data handling
- Stream NEVER stops.

## Gate 4: Assertions at Gold Batch (Batch)
- Validates before writing final star schema fact tables
- ROW_NUMBER dedup on business keys to remove Kafka redeliveries
- If validation fails → job crashes → fix and rerun

## Monitoring (Across All Layers)
- DLQ row count spike alerts (every 5 mins)
- Data freshness checks: MAX(ingest_ts) per layer, alert if > 5 min stale
- Row count reconciliation: Landing = Bronze = Silver good + Silver DLQ
